# 性能提升分析报告

## 优化概述

本次优化通过为 `isEleEllipsis` 函数添加记忆化缓存（memoization cache），显著减少了不必要的 DOM 操作和布局重排（reflow）。

## 性能瓶颈分析

### 优化前的问题

原始 `isEleEllipsis` 函数每次调用都会：

1. **创建 DOM 元素**：`document.createElement('em')`
2. **插入 DOM**：`ele.appendChild(childDiv)` - 触发布局重排
3. **两次 getBoundingClientRect 调用**：
   - `ele.getBoundingClientRect()` - 强制布局重计算
   - `childDiv.getBoundingClientRect()` - 再次强制布局重计算
4. **移除 DOM 元素**：`ele.removeChild(childDiv)` - 再次触发布局重排

**关键性能指标：**
- 每次调用触发 **2-3 次布局重排（Reflow）**
- 每次调用执行 **2 次 getBoundingClientRect()**（最昂贵的 DOM API 之一）
- 频繁的 DOM 操作（创建、插入、删除元素）

### 在表格场景中的影响

在使用 Table/ProTable 时，如果有多列开启 `ellipsis: true`：

**示例场景：100 行 × 10 列的表格**
- 初始渲染：1000 个单元格 = **1000 次完整检查**
- 每次滚动：可能触发所有可见单元格的重新检查
- 每次窗口调整：触发所有单元格的重新检查

## 优化方案

### 记忆化缓存策略

新实现使用 `WeakMap` 缓存检查结果，只在以下情况才重新计算：

1. **元素尺寸改变**：`width` 或 `height` 变化
2. **滚动尺寸改变**：`scrollWidth` 或 `scrollHeight` 变化
3. **子节点数量改变**：`childNodes.length` 变化

**缓存命中时的操作：**
- 1 次 `getBoundingClientRect()` 调用（仅用于比较）
- 5 次简单数值比较
- **0 次 DOM 操作**
- **0 次布局重排**

## 性能提升计算

### 单次调用性能对比

| 操作 | 优化前 | 优化后（缓存命中） | 提升 |
|------|--------|-------------------|------|
| DOM 元素创建 | 1 次 | 0 次 | ✅ 100% |
| DOM 插入/删除 | 2 次 | 0 次 | ✅ 100% |
| getBoundingClientRect | 2 次 | 1 次 | ✅ 50% |
| 布局重排 (Reflow) | 2-3 次 | 0 次 | ✅ 100% |

### 典型使用场景性能提升

#### 场景 1：表格初始渲染
**100 行 × 10 列 = 1000 个单元格**

- **首次渲染**（冷缓存）：
  - 优化前：1000 次完整检查
  - 优化后：1000 次完整检查（需要填充缓存）
  - **提升：0%**（符合预期，必须计算一次）

#### 场景 2：表格滚动（无尺寸变化）
**假设滚动显示 200 个新单元格，其中 150 个之前已渲染过**

- **优化前**：200 次完整检查
  - DOM 操作：400 次（创建+删除）
  - getBoundingClientRect：400 次
  - 布局重排：400-600 次

- **优化后**：
  - 缓存命中（150 个）：150 × 1 次 getBoundingClientRect = 150 次
  - 缓存未命中（50 个）：50 × 完整检查
  - 总 DOM 操作：100 次（仅新单元格）
  - 总 getBoundingClientRect：200 次（150 缓存检查 + 50 完整）
  - 布局重排：100-150 次（仅新单元格）

**性能提升：**
- DOM 操作减少：**75%** ↓
- getBoundingClientRect 减少：**50%** ↓
- 布局重排减少：**75-80%** ↓

#### 场景 3：频繁的 useEffect 触发（如鼠标悬停、内容更新等）
**同一元素在短时间内检查 10 次，尺寸未变化**

- **优化前**：10 次完整检查
  - DOM 操作：20 次
  - getBoundingClientRect：20 次
  - 布局重排：20-30 次

- **优化后**：1 次完整检查 + 9 次缓存命中
  - DOM 操作：2 次（仅首次）
  - getBoundingClientRect：10 次（1 次完整 + 9 次缓存检查）
  - 布局重排：2-3 次（仅首次）

**性能提升：**
- DOM 操作减少：**90%** ↓
- getBoundingClientRect 减少：**50%** ↓
- 布局重排减少：**90%** ↓

## 内存影响

### 缓存内存开销

每个缓存条目包含：
```typescript
{
  width: number,        // 8 bytes
  height: number,       // 8 bytes
  scrollWidth: number,  // 8 bytes
  scrollHeight: number, // 8 bytes
  children: number,     // 8 bytes
  isEllipsis: boolean   // 1 byte
}
```

**每个元素缓存开销：~41 bytes + WeakMap 开销**

**典型场景（1000 个单元格）：**
- 缓存内存：~41 KB
- WeakMap 开销：~20-30 KB
- **总计：~60-70 KB**

**优势：**
- 使用 `WeakMap` 自动垃圾回收，元素移除后缓存自动清理
- 相比性能提升，内存开销可忽略不计
